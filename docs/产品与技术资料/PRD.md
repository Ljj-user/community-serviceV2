# 社区公益服务对接管理平台 PRD v1.0

## 1. 产品定位

本系统面向社区公益服务场景，连接有服务需求的居民、愿意参与公益服务的志愿者，以及负责审核和管理的社区工作人员。

平台通过公益需求发布、志愿者认证、服务推荐、接单处理、评价反馈、数据统计和异常预警等功能，实现社区公益服务的线上化、流程化和可管理化。

一句话定位：

> 帮社区把“谁需要帮助”和“谁能提供帮助”对接起来。

## 2. 目标用户

### 2.1 居民用户

居民是平台的基础用户。

主要操作：

- 发布公益求助；
- 查看自己的求助进度；
- 确认服务完成；
- 评价志愿者；
- 查看社区公告；
- 查询便民信息；
- 申请成为志愿者。

### 2.2 认证志愿者

志愿者由居民申请认证后获得。

主要操作：

- 维护个人服务技能；
- 设置可服务时间；
- 查看推荐求助单；
- 接受公益服务任务；
- 更新服务进度；
- 查看历史服务记录。

### 2.3 重点关怀对象

重点关怀对象是需要社区长期关注的特殊居民。

包括：

- 独居老人；
- 残疾居民；
- 困难家庭成员；
- 长期病患；
- 其他需社区关注人员。

主要特点：

- 可正常发布求助；
- 可被管理员标记；
- 可纳入异常预警；
- 可绑定紧急联系人。

### 2.4 社区管理员

管理员负责平台后台管理。

主要操作：

- 审核用户认证；
- 管理重点关怀对象；
- 审核公益需求；
- 管理服务订单；
- 发布社区公告；
- 维护便民信息；
- 查看统计数据；
- 处理异常预警。

## 3. 设计原则

### 3.1 统一账号，多角色权限

移动端采用统一账号设计。用户基础身份为居民，用户可以发布求助，也可以在完成认证后成为志愿者。

系统不拆分“求助者账号”和“志愿者账号”。系统通过角色字段、认证状态和权限控制区分不同操作。

### 3.2 社区归属与数据边界

用户注册后默认为未加入社区状态。用户需要通过社区邀请码或管理员手动绑定的方式加入社区。

推荐流程为：

> 用户注册 → 输入社区邀请码 → 提交加入申请 → 管理员审核 → 绑定社区 → 访问本社区数据

社区绑定规则：

- 用户绑定社区后，系统记录用户的 `community_id`；
- 用户只能查看所属社区的公告、便民信息、公益需求和服务记录；
- 管理员只能管理自己负责社区范围内的数据；
- 老人或线下登记用户可由管理员手动绑定社区；
- 定位匹配社区不作为主流程，只作为后续可扩展能力。

### 3.3 公益服务优先

系统不做闲置交易、社区论坛、商业跑腿和在线支付。核心功能围绕公益需求对接、志愿者服务、重点对象关怀和后台管理展开。

### 3.4 管理闭环清晰

每一条公益需求都应具备明确状态。管理员可以审核、查看、统计和追踪服务过程。

## 4. 核心业务流程

平台核心流程为：

> 居民发布求助 → 管理员审核 → 系统推荐 → 志愿者接单 → 服务执行 → 居民确认 → 服务评价 → 数据沉淀 → 异常预警

该流程是系统设计和论文撰写的主线。

## 5. 功能模块

### 5.1 用户认证与重点关怀对象管理

该模块用于管理用户身份和服务权限。

功能包括：

- 居民注册登录；
- 用户基础信息维护；
- 志愿者认证申请；
- 管理员审核志愿者；
- 志愿者技能标签维护；
- 重点关怀对象标记；
- 紧急联系人维护；
- 认证状态管理。

核心规则：

- 普通居民可以发布求助；
- 只有认证志愿者可以接单；
- 重点关怀对象可被纳入异常检测；
- 管理员负责审核和标记。

### 5.2 公益需求发布与管理

该模块用于居民发布公益服务需求。

需求类型包括：

- 陪诊陪同；
- 代买取药；
- 家务协助；
- 情感陪伴；
- 应急帮助；
- 其他公益服务。

居民发布时填写：

- 服务类型；
- 服务时间；
- 服务地点；
- 联系方式；
- 需求描述；
- 是否紧急；
- 是否需要管理员优先处理。

需求状态包括：

- 待审核；
- 待接单；
- 已接单；
- 服务中；
- 待确认；
- 已完成；
- 已取消；
- 审核未通过。

### 5.3 公益服务对接管理

该模块负责需求和志愿者之间的匹配。

功能包括：

- 管理员审核需求；
- 系统推荐合适志愿者；
- 志愿者查看推荐单；
- 志愿者主动接单；
- 服务状态流转；
- 管理员查看服务进度。

推荐依据包括：

- 志愿者技能；
- 服务距离；
- 可服务时间；
- 历史接单类型；
- 服务评分；
- 服务完成率。

### 5.4 评价反馈管理

该模块用于服务完成后的质量反馈。

功能包括：

- 居民确认服务完成；
- 居民评价志愿者；
- 填写评分和文字反馈；
- 管理员查看评价；
- 异常评价进入后台提醒。

评价数据可用于：

- 志愿者信用参考；
- 推荐规则优化；
- 数据统计分析；
- 异常行为识别。

### 5.5 社区公告管理

该模块用于社区信息发布。

功能包括：

- 管理员发布公告；
- 管理员编辑公告；
- 管理员删除公告；
- 居民查看公告列表；
- 居民查看公告详情。

公告内容包括：

- 公益活动通知；
- 志愿者招募；
- 社区政策；
- 安全提醒；
- 服务安排。

### 5.6 便民信息管理

该模块用于维护社区常用服务信息。

功能包括：

- 管理员新增便民信息；
- 管理员修改信息；
- 管理员删除信息；
- 居民分类查询。

便民信息类型包括：

- 社区服务电话；
- 附近医院；
- 药店信息；
- 维修服务；
- 政务窗口；
- 公益组织联系方式。

### 5.7 数据统计与异常预警

该模块用于后台数据分析和风险提醒。

统计内容包括：

- 公益需求总数；
- 已完成服务数；
- 服务完成率；
- 服务类型占比；
- 志愿者数量；
- 活跃志愿者排行；
- 重点关怀对象数量；
- 异常预警数量。

异常检测规则包括：

- 重点对象连续多日未登录；
- 重点对象长期无求助记录；
- 某区域短时间求助量上升；
- 某用户频繁发布紧急求助；
- 志愿者多次接单后取消；
- 服务评价连续偏低。

预警处理流程：

> 系统生成预警 → 管理员查看 → 网格员跟进 → 填写处理结果 → 预警关闭

## 6. AI 辅助功能

AI 功能不作为主业务模块，而是作为辅助能力。

居民端 AI 辅助：

- 帮助用户整理求助描述；
- 回答常见公益服务问题；
- 降低老人填写难度。

管理端 AI 分析：

- 总结异常预警原因；
- 分析近期服务趋势；
- 给管理员提供处理建议。

论文中建议表述为：

> AI 功能主要用于辅助信息填写与后台分析，不替代人工审核和实际决策。

## 7. 权限设计

系统采用统一账号和多角色权限设计。

用户基础身份为居民，可扩展为：

- 普通居民；
- 认证志愿者；
- 重点关怀对象；
- 管理员。

| 功能 | 普通居民 | 认证志愿者 | 重点对象 | 管理员 |
| --- | --- | --- | --- | --- |
| 发布求助 | 是 | 是 | 是 | 否 |
| 接受服务单 | 否 | 是 | 视情况 | 否 |
| 查看公告 | 是 | 是 | 是 | 是 |
| 查看便民信息 | 是 | 是 | 是 | 是 |
| 申请志愿者 | 是 | 已认证 | 可申请 | 否 |
| 审核需求 | 否 | 否 | 否 | 是 |
| 管理用户 | 否 | 否 | 否 | 是 |
| 查看统计 | 否 | 否 | 否 | 是 |
| 处理预警 | 否 | 否 | 否 | 是 |

## 8. 非功能需求

### 8.1 易用性

移动端操作要简单，入口应清晰。

重点入口包括：

- 我要发布求助；
- 我去帮助别人；
- 我的求助；
- 我的服务；
- 志愿者认证。

### 8.2 安全性

- 用户登录后才能发布求助；
- 志愿者认证后才能接单；
- 管理员审核关键数据；
- 敏感信息后台可控；
- 重点对象信息不得公开展示。

### 8.3 可维护性

- 服务类型可配置；
- 公告可后台维护；
- 便民信息可后台维护；
- 异常规则可扩展；
- 数据统计可按时间筛选。

## 9. 系统边界

本系统不做以下功能：

- 不做闲置交易；
- 不做社区论坛；
- 不做即时社交广场；
- 不做商业跑腿平台；
- 不做在线支付；
- 不做复杂地图调度。

系统重点是：

> 公益需求对接、志愿者服务管理、重点对象关怀和后台管理分析。
