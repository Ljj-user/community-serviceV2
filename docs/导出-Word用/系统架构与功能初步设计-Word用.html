社区公益服务对接管理平台 — 系统架构与功能初步设计

本文突出总体结构、角色分工与核心能力，省略实现细节与接口清单，便于开题/中期/论文「总体设计」章节引用。

1. 建设目标（一句话）

搭建 Web 管理端 + 移动端 与 统一后端，支撑居民发布需求、志愿者认领服务、社区审核与网格化隔离，并形成可演示的积分/时间币与评价闭环。

2. 系统架构图

2.2 业务模块拓扑（能力视图）

认证与权限 ──┬──► 网格/社区 ──► 需求 lifecycle ──► 大厅与看板
            └──► 公告
待办 / 点赞 / 审计 / 配置（辅助）

要点：需求（REQ）与网格（GRID）是主线；其余模块增强可演示性与管理完整性。

flowchart LR
  AUTH[认证与权限] --> GRID[网格/社区]
  AUTH --> REQ[需求]
  GRID --> REQ
  REQ --> HALL[大厅与看板]
  AUTH --> ANN[公告]

3. 技术选型（摘要）

层次技术作用
管理端Vue 3、Naive UI、Vite后台配置、用户与审核、统计看板
移动端Vue 3、NutUI、Vite居民/志愿者场景、服务大厅
后端Spring Boot 3、Spring Security、JWT、MyBatis-Plus接口、鉴权、持久化
数据库MySQL 8业务数据与审计

4. 角色与功能初步设想（重点）

角色核心诉求主要功能（初稿）
超级管理员全局治理系统参数、高危导出/备份、跨社区监管（按现有权限划分）
社区管理员本网格运营需求审核（通过/驳回）、公告发布、志愿者/居民相关管理、数据看板
居民老人获得帮助注册/登录、绑定社区、发布需求、查看进度、服务完成后评价
志愿者接单服务浏览已发布需求、认领、提交服务时长、查看评价与个人大厅数据

业务闭环（论文可强调）：发布 → 审核通过 → 发布展示 → 认领 → 完成 → 评价；社区维度通过 community_id / sys_region 做数据隔离与展示（省、市字段用于增强真实感，不参与核心鉴权规则）。

5. 非功能与安全（点到为止）

  • 认证：无状态 JWT；角色区分超管 / 社区管理员 / 普通用户（居民与志愿者通过身份字段区分）。
  • 隔离：需求与用户绑定社区，列表与统计在业务上按社区过滤（具体规则以代码为准）。
  • 可演进：密码算法、短信、地图选址等可在「展望」中表述为后续工作。
