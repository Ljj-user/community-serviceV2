# 需求可行性与优先级分析（含实现情况）

> 用途：该表用于论文/开题/答辩 PPT 的“需求分析”部分，可直接在编辑器或 Typora 中打开截图。
>
> 字段说明：**可行性**综合考虑技术成熟度、开发成本、实现风险与维护复杂度；**优先级**按对核心业务的影响（高/中/低）排序；**实现状态**区分已实现与开题报告中提出但当前未实现的内容。

| 需求名称 | 简要说明 | 可行性分析（技术 + 成本 + 风险） | 优先级 | 实现状态 |
|---|---|---|---|---|
| 用户注册登录与身份认证 | 支持居民、志愿者、管理员账号登录，基础认证能力 | 认证方案成熟（Spring Security/JWT 等），实现成本低；风险主要在密码/令牌安全与过期策略，可通过规范配置与审计降低 | 高 | 已实现 |
| 角色与权限管理 | 多角色分权（区/街道/社区管理员、志愿者等），控制菜单/接口权限 | 基于角色字段与接口鉴权拦截即可扩展；开发成本中等，安全收益高；风险在权限边界不清，可通过权限矩阵与测试用例控制 | 高 | 已实现 |
| 服务需求发布与审核 | 居民发布服务需求，管理员审核上架/驳回，状态流转 | 表结构与流程清晰，按“状态机字段 + 审核记录”实现可控；成本中等；风险在重复提交、撤回、越权操作等边界条件 | 高 | 已实现 |
| 志愿者认领与服务记录 | 志愿者认领需求，记录服务时长与完成情况 | 业务模型常见（认领表/时长表），技术可行性高；成本中等；风险在并发认领与重复认领，可用事务/唯一约束/乐观锁处理 | 高 | 已实现 |
| 服务评价与反馈 | 服务完成后评价打分与文字反馈 | 评价模块实现成熟，成本不高；风险在内容合规与恶意评论，可用敏感词过滤、举报/屏蔽策略降低 | 中 | 已实现 |
| 社区公告管理 | 公告发布、列表、详情展示 | 典型 CRUD，开发成本低；风险低（主要是内容审核与权限） | 中 | 已实现 |
| 区域网格管理（区-街道-社区 + 省市） | 维护区域树，用户归属社区/网格绑定 | 自关联树结构成熟，开发成本较低；风险主要在数据维护与一致性校验，可通过后台校验与导入脚本降低 | 高 | 已实现 |
| 个人待办与任务提醒 | 待办分组、任务列表，用于日常业务跟进 | 任务管理属于通用功能，技术可行；成本中等；风险在与业务流程耦合度高，适合先做通用待办再逐步接入业务触发 | 中 | 已实现 |
| 数据备份与导出记录 | 记录备份/导出操作、时间、操作者等元数据 | 记录层实现简单；真正的备份/导出涉及权限与文件存储规范，成本中等；风险在敏感数据泄露与越权导出 | 中 | 已实现 |
| 点赞与支持统计 | 对公告/内容点赞计数，提高互动 | 计数与防刷策略成熟；成本低；风险在刷赞，可通过登录限制与频控降低 | 低 | 已实现 |
| 统计报表与可视化分析 | 服务次数、时长、评价等统计图表与导出 | 技术可行（聚合查询 + ECharts/报表导出），但工作量较大；风险在数据口径一致性与性能（大数据量聚合） | 中 | 未实现（开题报告中提出） |
| 智能匹配与推荐（志愿者 ←→ 需求） | 基于技能/标签/历史记录推荐志愿者或需求 | 可用规则引擎/简单推荐先落地；但需要额外建模与调参，效果不确定；成本较高、测试难度较大 | 中 | 未实现（开题报告中提出） |
| 消息通知（短信/微信/邮件） | 审核结果、认领成功、服务提醒等主动通知 | 对接第三方平台路径清晰，但涉及费用、平台审核与运维；风险在合规与误发、号码泄露等，需要完善权限与审计 | 中 | 未实现（开题报告中提出） |
| 志愿者信用与等级体系 | 依据次数/时长/评价形成信用分与等级徽章 | 技术实现不难，但规则设计偏管理制度；风险在指标不合理引发争议与“刷分”，需要明确奖惩与防作弊 | 低 | 未实现（开题报告中提出） |
| 满意度调查与问卷 | 定期问卷收集使用反馈与服务满意度 | 可自建问卷表或接入第三方；技术风险低，但会增加运营成本；更偏论文加分项 | 低 | 未实现（开题报告中提出） |
| 系统运维监控与告警 | 错误率、响应时间、资源监控与告警 | 技术可行（Prometheus/Grafana 等），但部署与维护成本较高；对毕业设计加分但非核心交付；风险在环境限制与运维复杂度 | 低 | 未实现（开题报告中提出） |
